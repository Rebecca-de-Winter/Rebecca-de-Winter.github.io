Two practical sessions

What attendees will take away

These talks are for developers who want their interfaces to feel clearer for users and calmer to maintain in the codebase. Both sessions are built around practical decision-making: how do we guide people through a product, and how do we choose the right layer for the job?

Talk One

How Thinking Like a Teacher Helps You Build Better Software

Stop making users guess the rules.

Every product teaches people how to use it, whether we plan for that or not. A confusing form, vague error, empty dashboard, hidden permission rule, or unclear workflow can make users feel like they have failed, when the product simply has not taught them what it expects.

This talk shows developers how to design software that explains itself: clearer expectations, better feedback, fewer avoidable errors, and more supportive pathways through complex features.

  • Design forms that guide instead of punish
  • Write feedback that helps users recover, not just fail
  • Turn dashboards into clear next-action spaces
  • Make frontend and backend rules teach the same version of the product

Talk Two

Don’t Fear the CSS: How to Build Better Interfaces with Less JavaScript

Untangle your frontend before it turns into spaghetti.

Modern interfaces do a lot: they resize, reveal, respond, animate, validate, display status, and guide people through workflows. It is easy for all of that behaviour to pile into components until layout, styling, visual state, and application logic are all fighting for space.

This talk helps developers choose the right layer for each kind of behaviour, using HTML and CSS more deliberately so JavaScript can stay focused on the work that genuinely needs logic, data, coordination, or backend truth.

  • Spot when components are doing work the browser can already handle
  • Use CSS for layout, visual state, responsiveness, and motion
  • Keep JavaScript focused on data, permissions, APIs, and business rules
  • Build interfaces that are easier to read, debug, and safely change

Hi, I’m Becky Cole. I’m a software developer with a background in education, digital learning, mentoring, and media.

My talk is called “How Thinking Like a Teacher Helps You Build Better Software.”

The core idea is simple: every product teaches people how to use it, whether we plan for that or not.

A confusing form, vague error message, empty dashboard, hidden permission rule, or unclear workflow can make users feel like they have failed, when the product has not taught them what it expects and how to succeed.

In education, we think carefully about concepts like scaffolding, feedback, sequencing, cognitive load, and clear success criteria. Those same ideas are incredibly useful in software development, because users are learning our systems when they use them.

This talk guides developers on how to design software that takes a learner's point of view: forms that guide instead of punish, feedback that helps users recover and get back to safety, dashboards built around what happens next, and frontend and backend rules that teach one clear version of how the product works.

Attendees will leave with a practical toolkit and a new way to see their work: every form, error, dashboard, permission rule, and workflow is a teaching moment. When we design those moments deliberately, software can do more than reduce mistakes. It can help people feel capable, confident, and clear about what to do next and your website will be more successful!

Proposal 02

Don’t Fear the CSS: How to Build Better Interfaces with Less JavaScript

This talk is for developers whose components have started collecting every responsibility in the room: layout, styling, animation, visual state, open-and-closed behaviour, conditional classes, business rules, and API-driven logic.

It shows how to sort that tangle into clearer layers: what belongs in HTML, what CSS can handle beautifully, what genuinely needs JavaScript, and when a framework or library is worth reaching for.

Read transcript

Hi, I’m Becky Cole, and my talk is called “Don’t Fear the CSS: How to Build Better Interfaces with Less JavaScript.”

When I started building full-stack React projects, I often reached for JavaScript whenever an interface needed to do something. If something opened, changed, moved, responded, adapted, or looked different in a new state, my instinct was often to put that behaviour into a component's JSX.

The thing is, the more I built, the more I noticed that some of my components were becoming harder to manage because layout, styling, animation, visual state, interaction behaviour, and application logic were all piling into the same layer.

Modern interfaces can now do a lot. They resize, reveal, respond, animate, validate, display status, and guide people through workflows. But that can get overwhelming quickly. The useful question is not “Can JavaScript do this?” The better question is: which layer should own this behaviour?

JavaScript and frameworks are still essential for data fetching, routing, authentication, permissions, saved choices, API calls, and complex state. But many interface behaviours can live closer to the browser, in HTML and CSS.

This talk gives developers a decision framework for separating interface behaviour from application behaviour, using CSS as part of frontend architecture, and keeping JavaScript for the work that truly needs logic, data, coordination, or backend truth. The goal is clearer interfaces and code that is easier to read, debug, and change. And can save you a lot of headaches!
