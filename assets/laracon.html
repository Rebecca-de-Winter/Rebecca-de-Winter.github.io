Proposal 01

How Thinking Like a Teacher Can Help You Build Better Software

This talk connects my education background with practical software design decisions, especially around clarity, guidance, confidence, and reducing cognitive load for users.

Read transcript

Hi, I’m Rebecca Cole. I’m an early-career software developer, but before moving into tech I spent years working in education, digital learning, mentoring, and community programs.

My proposed talk is called “How Thinking Like a Teacher Can Help You Build Better Software.”

The idea is simple: good software does more than function. It helps people understand what to do next.

When I started building full-stack projects, I realised that a lot of the skills I had used as an educator still mattered in software. Things like sequencing information, reducing cognitive load, giving people clear first steps, designing forgiving forms, using consistent language, and helping users recover when they feel unsure.

In this talk, I’ll use examples from real projects I’ve built, including PodFlow and Backyard Festival, to show how a teaching mindset can improve onboarding, empty states, form design, status messages, dashboards, permissions, and workflow design.

This is not a theory talk. It’s a practical talk for developers who want their interfaces to feel clearer, kinder, and easier to use.

I want attendees to walk away with a simple question they can apply immediately: if this interface were teaching the user, what would it explain first?

Proposal 02

Don’t Fear the CSS: How to Build Better Interfaces with Less JavaScript

This talk is about keeping frontend code easier to understand by giving layout, visual state, responsiveness, and small interactions to the right layer.

Read transcript

Hi, I’m Rebecca Cole, and my proposed talk is called “Don’t Fear the CSS: How to Build Better Interfaces with Less JavaScript.”

When I started building full-stack React projects, I often reached for JavaScript whenever an interface needed to do something. If something opened, changed, responded, moved, adapted, or looked different in a new state, my instinct was often to put that behaviour into a component.

But over time, I started noticing that some of my components were becoming harder to understand because layout, styling, animation, visual state, and application logic were all piling into the same place.

This talk is about choosing the right layer for the job.

JavaScript and frameworks are absolutely essential for things like data fetching, routing, authentication, permissions, and complex state. But many interface behaviours can be handled more clearly with HTML and CSS.

Using examples from real frontend projects, I’ll show practical ways developers can use CSS for layout, responsiveness, visual states, transitions, progressive enhancement, and small interaction patterns without overloading their JavaScript.

The goal is not to avoid JavaScript. The goal is to use it where it belongs, and let the browser do the work it already knows how to do.
